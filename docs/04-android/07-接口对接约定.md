## 一、摄像头输入约定（Android → 第 3 部分感知）

> 所有默认值的**唯一真相源**是代码里的 `CameraInputSpec`，调参只改那里，文档随之更新。

| 项 | 约定值 | 说明 / 责任方 |
| --- | --- | --- |
| **格式 format** | `YUV_420_888` | CameraX `ImageAnalysis` 原生输出，零拷贝最省。若模型需 `NV21`/`RGB`，**由算法侧转换** |
| **分辨率 resolution** | `640 × 480`（= `width × height`） | 对 CPR 姿态识别足够、延迟低；上限可放宽到 720p，可配置 |
| **帧率 frame rate** | 采集 30fps，**分析节流 ~15fps** | 按压约 2Hz，15fps 足够；用 `KEEP_ONLY_LATEST` 丢旧帧防堆积 |
| **rotation** | `imageInfo.rotationDegrees`（0/90/180/270） | Android 透传，**由算法侧据此旋正**，Android 不预旋转 |
| **timestamp** | `SystemClock.elapsedRealtime()`（单调时钟，**毫秒**） | 禁止混用 wall-clock，避免对时漂移；字段 `timestampMs` |
| **session_id** | Android 会话开始生成 UUID，经 `start(sessionId)` 下发 | 感知源须**回填到每条 `PerceptionEvent.sessionId`**；帧元数据 `FrameMeta.sessionId` 同源 |

> 关于 resolution：契约里**不单设 resolution 字段**，因为 `FrameMeta.width × height` 就是分辨率，
> 单设会和宽高重复、易不一致。需要可读标签时用 `CameraInputSpec.resolutionLabel`（如 `"640x480"`）。

### 代码对应

```kotlin
// 推荐默认规格，联调直接用
val spec = CameraInputSpec.DEFAULT
// spec.analysisWidth=640, analysisHeight=480, captureFps=30, analysisFps=15,
// pixelFormat="YUV_420_888", keepOnlyLatest=true
```

```kotlin
data class FrameMeta(
    val width: Int, val height: Int,   // 分辨率
    val rotationDegrees: Int,          // 0/90/180/270
    val format: String,                // YUV_420_888
    val timestampMs: Long,             // 单调时钟 ms
    val sessionId: String,             // 会话 id
)
```

### 关于按压深度（已确认不提供）

v1 明确：**不承诺精确按压深度(cm)，只做视觉代理评分**，因此 `PerceptionEvent` **不含 depth 字段**。
对应地，Android 仪表盘的第二格已从“深度”改为 **“手臂(arm_straight)”**，
其余质量信息体现在 `qualityScore`（含 v1 的 continuity/posture 权重）。
若后续第 3 部分要补深度代理分，走 `PerceptionEvent.rawSignals`（口袋字段）即可，无需改接口。

## 二、联调传输方式

> **结论：走进程内 Kotlin 对象（接口 + Flow），不走 HTTP/JSON。**

理由：本产品是**端侧、离线、同一 App 进程**的 Edge AI（见分工表第 2 部分“端侧推理、离线”）。
同进程直接传对象最快、无序列化开销，也最简单。

| 通道 | 用途 | 形式 |
| --- | --- | --- |
| **运行时（默认）** | 感知 → Android → Agent | 进程内对象：`Flow<PerceptionEvent>` / `Flow<GuidanceAction>` |
| 联调阶段 | 链路连通性验证 | 先用 `mock/` 下的 Mock 实现，再逐个换真实（改 `ServiceLocator` 一行） |
| 可选 · 调试 | 录制 / 回放 / 复盘 | JSON 落盘（仅离线日志，**不作运行时通道**） |
| 仅当跨进程 | 若某部分是独立进程/服务 | 才需 HTTP/JSON，请提前告知，另行约定 schema |

### 实现要求

- 第 3 部分：实现 `CprPerceptionSource`（即使底层是 native/.so，也包一层 Kotlin 类）。
- 第 2 部分：实现 `GuidanceAgent`。
- 接入只改 `ServiceLocator`：

```kotlin
ServiceLocator.perceptionFactory = { ctx -> RealPerceptionSource(ctx) } // 第3部分
ServiceLocator.agentFactory      = { ctx -> GemmaGuidanceAgent(ctx) }   // 第2部分
```

## 三、session_id 贯通链路（已落地）

```
Android 会话开始: sessionId = UUID
   │
   ├─▶ perceptionSource.start(sessionId)   感知源回填到每条 PerceptionEvent.sessionId
   ├─▶ agent.onSessionStart(sessionId)      Agent 按会话维护状态
   └─▶ 会话结束 SessionLog.sessionId 同源 ─▶ HandoverReport.sessionId
```

同一次急救的所有帧、感知事件、指导、日志、报告共用一个 id，便于对齐与回放。
