## 一、工程级结构（Gradle 配置）

```
cpr_new/
├── settings.gradle.kts          # 声明只有一个模块 :app
├── build.gradle.kts             # 顶层：插件版本声明（不放具体配置）
├── gradle/libs.versions.toml    # 版本目录：统一管理所有依赖版本（新增了协程/ViewModel）
├── app/build.gradle.kts         # 模块配置：SDK 版本、依赖列表
├── app/src/main/AndroidManifest.xml  # 权限声明 + 入口 Activity（相机/录音/定位/震动权限）
├── docs/04-android/             # 第 4 部分的全部说明文档
└── app/src/main/java/com/example/cpr_new/   # 全部业务代码
```

## 二、代码分层（核心）

整体是一条**单向依赖**链：`UI → 编排 → 硬件/实现 → 契约`。
下层不知道上层存在，所以上层可以随意替换实现。

```
com/example/cpr_new/
│
├── MainActivity.kt              【入口】装配依赖 + 注入 ViewModel + 触发权限请求
│
├── core/                        【地基：纯 Kotlin，无 Android UI 依赖】
│   ├── contract/                ★ 与第2、3部分对接的"协议"（最关键，解耦核心）
│   │   ├── PerceptionEvent.kt   　第3部分给我的：手位/频率/深度/置信度
│   │   ├── CprPerceptionSource.kt 第3部分要实现的接口（感知源）
│   │   ├── GuidanceAction.kt    　第2部分给我的：播报文本/大字/震动/阶段
│   │   ├── GuidanceAgent.kt     　第2部分要实现的接口（决策 Agent）
│   │   └── SessionModels.kt     　会话日志 / 交接报告 / 定位快照
│   └── di/                      手写依赖注入
│       ├── CprDependencies.kt   　把所有能力打成一个包
│       └── ServiceLocator.kt    ★ 一处切换 Mock ↔ 真实实现
│
├── hardware/                    【硬件适配层：纯系统 API，零重依赖，全部容错降级】
│   ├── audio/TtsController.kt           语音播报
│   ├── audio/AudioRecorderController.kt 录音留证
│   ├── haptics/HapticController.kt      震动反馈
│   ├── location/LocationProvider.kt     GPS 定位
│   └── permission/RuntimePermissions.kt 运行时权限（Compose 友好）
│
├── feature/                     【业务编排层：唯一懂"流程"的地方】
│   ├── session/
│   │   ├── CprSessionViewModel.kt  ★ 总线：串联 感知→Agent→硬件→UI
│   │   ├── CprSessionState.kt      UI 的唯一数据源（状态快照）
│   │   ├── Metronome.kt            节拍器（100-120次/分，可热变速）
│   │   └── QualityScorer.kt        质量分聚合（仅展示用，非医学标准）
│   └── emergency/EmergencyDialer.kt 120 拨号（安全的 ACTION_DIAL）
│
├── ui/                          【界面层：Compose，无状态，只读状态+抛回调】
│   ├── EmergencyPalette.kt          急救高对比配色
│   ├── component/CprComponents.kt   大字卡片/仪表盘/节拍脉冲/告警横幅
│   ├── screen/CprGuidanceScreen.kt  主屏（待命态 + 进行态 + 报告弹窗）
│   └── theme/                       原有 Compose 主题（保留）
│
└── mock/                        【演示桩：真实实现接入后可删】
    ├── MockCprPerceptionSource.kt   假装是第3部分，吐模拟感知数据
    └── MockGuidanceAgent.kt         假装是第2部分，用简单规则给指导
```

## 三、怎么理解这套结构（打比方）

- **`core/contract` 是合同**：规定“第 3 部分给我什么、第 2 部分给我什么”。只要别人按合同实现，本模块一行不用改。
- **`mock` 是替身演员**：真演员（真实模型）到场前先顶上，保证戏能从头演到尾（断网/无模型也能跑）。
- **`ServiceLocator` 是导演**：决定这场戏用替身还是真演员，换人只改这一个文件两行。
- **`CprSessionViewModel` 是总线**：感知数据 → 喂 Agent → 拿指导 → 分发给语音/震动/节拍/界面，全在这里编排。
- **`ui` 只负责好看**：不含任何逻辑，纯粹把 `CprSessionState` 画出来，方便随时改版。

## 四、一次急救的数据流

```
摄像头/Mock ──感知事件──▶ ViewModel ──喂──▶ Agent
                            │              │
                     更新质量分/UI    返回指导动作
                            │              │
                            ▼   ┌──────────┴──────────┐
                          界面  语音播报 震动 节拍变速 切阶段
                            │
              结束 ──▶ 打包会话日志 ──▶ Agent生成交接报告 ──▶ 弹窗展示
```

---

## 五、各层职责一句话总结

| 层 | 目录 | 一句话职责 | 改动频率 |
| --- | --- | --- | --- |
| 入口 | `MainActivity.kt` | 装配 + 注入 | 低 |
| 契约 | `core/contract` | 三方对接协议 | 极低（改了影响所有人） |
| 注入 | `core/di` | 选实现 | 接入真实模型时改 |
| 硬件 | `hardware` | 封装系统能力 | 低 |
| 编排 | `feature/session` | 串流程 | 中 |
| 拨号 | `feature/emergency` | 120 | 低 |
| 界面 | `ui` | 渲染 | 高（改版） |
| 桩 | `mock` | 离线演示 | 真实实现后可删 |
